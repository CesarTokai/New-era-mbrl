ENCONTRANDO LAS DIFERENCIAS REALES:
═══════════════════════════════════════════════════════════════════════════════════════════

✅ DIFERENCIAS SIGNIFICATIVAS:

1. PERMISOS DE ACCESO
   ❌ /furniture:
      - GET / → SIN @PreAuthorize (PÚBLICO)
      - POST  → @PreAuthorize("hasRole('ADMIN')") - SOLO ADMIN exacto

   ✅ /api/products:
      - GET  → @PreAuthorize("hasAnyRole('USER', 'ADMIN')") - Más flexible
      - POST → @PreAuthorize("hasAnyRole('USER', 'ADMIN')") - Acepta USER o ADMIN

2. ENDPOINTS ADICIONALES
   ✅ /api/products TIENE:
      - GET /api/products/low-stock → Productos con stock bajo
      - GET /api/products/{id}/related → Productos relacionados

   ❌ /furniture NO TIENE estos endpoints

3. PROPÓSITO DOCUMENTADO
   FurnitureController.java línea 19-21:
   /**
    * Controlador que expone los productos bajo la ruta /furniture/
    * para compatibilidad con el frontend existente.
    */

   → Fue creado POR COMPATIBILIDAD HISTÓRICA CON EL FRONTEND

═══════════════════════════════════════════════════════════════════════════════════════════
POR QUÉ EXISTEN AMBAS:
═══════════════════════════════════════════════════════════════════════════════════════════

ESCENARIO MÁS PROBABLE:

1. FASE INICIAL:
   - Se creó FurnitureController (/furniture) para el frontend
   - El frontend fue diseñado esperando esta ruta
   - Se estructura como un "alias" del concepto "muebles"

2. FASE POSTERIOR (REFACTORIZACIÓN):
   - Se creó ProductController (/api/products) como API "profesional"
   - Sigue convención RESTful (URLs comienzan con /api/)
   - Incluye endpoints más avanzados (low-stock, related)
   - Tiene permisos más granulares (acepta USER además de ADMIN)

3. RESULTADO:
   - AMBAS existen porque:
     a) El frontend sigue usando /furniture (legado)
     b) Nadie eliminó FurnitureController para evitar romper el frontend
     c) ProductController es la "versión nueva" más completa

¿POR QUÉ FALLA /furniture CON 404?
═══════════════════════════════════════════════════════════════════════════════════════════

❌ PROBLEMA REAL IDENTIFICADO:

El frontend intenta:
- GET http://localhost:8080/furniture/ → 404 Not Found

Pero en el código:
- FurnitureController.java EXISTE
- @RequestMapping("/furniture") ESTÁ DEFINIDO
- @GetMapping NO tiene @PreAuthorize (debería ser público)

¿POR QUÉ 404 SI EL CONTROLLER EXISTE?

RESPUESTA: El servidor podría estar:
1. Usando una versión compilada ANTIGUA sin FurnitureController
2. El controller NO está siendo escanteado por Spring (ver target/classes)
3. FurnitureController está comentado o deshabilitado
4. El jar/war deployado no incluye cambios recientes

COMPROBACIÓN:
- En target/classes/com/mx/mbrl/controller/ ¿existe FurnitureController.class?

PREGUNTA: ¿Por qué 2 APIs idénticas?
RESPUESTA: NO son completamente idénticas:
           - /furniture es LEGADO (por compatibilidad frontend)
           - /api/products es NUEVA (más completa y profesional)

           - El frontend usa /furniture pero está retornando 404
           - Probablemente porque el servidor compilado NO incluye FurnitureController
           - O está deshabilitado/comentado

           ✅ Usar /api/products (cambiar frontend de /furniture → /api/products)
           ✅ Eliminar FurnitureController para evitar confusión
           ✅ Centralizar toda la lógica en ProductController
