HALLAZGOS DE LA AUDITORÍA
═══════════════════════════════════════════════════════════════════════════════════════════

### ✅ PUNTOS POSITIVOS:

1. **Sin duplicación crítica**
   - Cada controller tiene responsabilidad única y clara
   - No hay 2 controllers haciendo lo mismo

2. **Convención de rutas consistente**
   - `/api/*` para API moderna
   - `/furniture/*` para conceptos de negocio específicos
   - Está bien definido

3. **Separación de responsabilidades**
   - AuthController ≠ CustomerController (no mezclan conceptos)
   - ReportController separado de ProductController
   - InventoryController independiente

4. **Permisos bien asignados**
   - @PreAuthorize usa hasAnyRole (flexible)
   - Algunos endpoints públicos donde es necesario
   - Consistencia en restricciones

### ⚠️ PUNTOS A OBSERVAR (no son críticos):

1. **InventoryController y ReportController usan @PreAuthorize a nivel de clase**
   ```java
   @RestController
   @PreAuthorize("hasAnyRole('USER', 'ADMIN')")
   public class InventoryController { ... }
   ```
   Esto significa que TODOS los endpoints requieren autenticación.
   - ✅ Correcto: Estos recursos son sensibles

2. **Rutas inconsistentes en formato**
   - `/api/products` (sin "s" al final en ruta base)
   - `/api/customers` (con "s")
   - `/api/inventory` (singular)
   - `/api/reports` (plural)
   - Esto es MENOR, pero se podría estandarizar
